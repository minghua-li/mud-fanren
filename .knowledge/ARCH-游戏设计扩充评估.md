## 二、技术可行性评估

### 2.1 目录结构方案

当前 `02-扩充内容/` 为单层目录。若将所有维度的游戏设计文件都堆入此目录，文件数量会膨胀至 10-15 个，不利于维护。

**推荐方案：02-扩充内容/ 下按领域分子目录**

```
.knowledge/
├── overview/                     # 总览（已有）
├── world/                        # 世界观（已有）
│   └── (新增) 02-区域游戏玩法.md
├── cultivation/                  # 修炼（已有）
│   └── (新增) 02-灵根养成与突破.md
├── factions/                     # 势力（已有）
│   └── (新增) 02-声望与互动玩法.md
├── items/                        # 物品（已有）
│   └── (新增) 02-丹药配方与炼制.md
│   └── (新增) 02-法宝进阶与养成.md
├── combat/                       # 战斗（已有）
│   └── (新增) 02-战斗机制与平衡.md
├── quests/                       # 任务（已有）
│   └── (新增) 02-任务链与奖励曲线.md
└── 02-扩充内容/                  # （保留现有 3 个跨领域文件）
    ├── 02-经济与资源.md          # 已有（跨领域：items/ + quests/）
    ├── 02-日常玩法.md            # 已有（跨领域：cultivation/ + combat/ + world/）
    └── 02-配角与势力.md          # 已有（跨领域：factions/ + world/）
```

**理由：**
1. 每个子目录已有剧本提取文档，游戏设计文件就近放置，便于对照阅读
2. 文件数量可控（每领域 1-2 个补充文件）
3. 新命名规范统一：`02-<主题>.md` 表示「游戏机制设计」（与已有 `1A-`、`1B-` 等剧本提取命名区分）
4. 现有 3 个跨领域文件保留在 02-扩充内容/ 根目录，它们跨越多个领域，不宜归入单一子目录

**备选方案 A：所有游戏设计文件继续平铺在 02-扩充内容/**
- 优点：改动最小
- 缺点：10-15 个文件同目录，管理成本高

**备选方案 B：每个维度创建平行目录**
- 例如 combat_game/, cultivation_game/ 等
- 优点：泾渭分明
- 缺点：目录数量翻倍，用户 `ls .knowledge/` 眼花缭乱

### 2.2 文件命名与引用规范

| 类型 | 命名模式 | 示例 | INDEX.md 中标识 |
|------|---------|------|----------------|
| 剧本提取 | `1A-` `1B-` 前缀 | `1A-人界地理.md` | 无特殊标记 |
| 游戏机制设计 | `02-` 前缀 | `02-区域游戏玩法.md` | ✨ 扩充内容 标记 |

`[[slug]]` 交叉引用规则维持不变，新增文件在 `RELATIONS.md` 中加入关联边。

## 四、推荐技术路线

### 4.1 分批实施（推荐）

考虑到用户催促「所有的要素都要」，且内容总量约 3000-5000 行，建议分 **3 批** 实施：

**第一批（P0，用户直接点名，最急迫）：**
- `items/02-丹药配方与炼制.md`  — 基于 1E 的丹药/材料数据
- `items/02-法宝进阶与养成.md` — 法宝升级/灵宝认主/青竹蜂云剑路线
- `cultivation/02-功法修炼机制.md` — 修炼进度/功法获取/冲突规则
- 同步更新 INDEX.md + RELATIONS.md

工作量预估：~1,500 行（3 个文件 × ~500 行），建议 2 名 worker 并行

**第二批（P1，游戏核心玩法体系）：**
- `combat/02-战斗机制与平衡.md` — PVE/PVP/五行克制/阵法
- `cultivation/02-灵根养成与突破.md` — 灵根洗练/突破玩法
- `world/02-区域游戏玩法.md` — 区域解锁/各区域特色
- `factions/02-声望与互动玩法.md` — 声望/门派/种族
- 同步更新 INDEX.md + RELATIONS.md

工作量预估：~1,500 行（4 个文件 × ~350-400 行），2-3 名 worker 并行

**第三批（P2，收尾与精细化）：**
- `quests/02-任务链与奖励曲线.md` — 奖励数值/LPC 实现
- 补充 02-经济与资源.md 的经济平衡框架
- 更新 INDEX.md、RELATIONS.md、1A-游戏要素总览.md

工作量预估：~500 行，1 名 worker

### 4.2 纯并行方案（备选，不推荐）

所有 8-10 个文件同时开写。优点是快，但缺点明显：
- 文件间强关联（经济/日常/法宝/丹药 相互引用），并行容易产生不一致
- 先写的 02-扩充内容/ 3 个文件是后续工作的重要参考——后写方向需要学习其方法
- 并行修改 INDEX.md + RELATIONS.md 合并冲突风险高

### 4.3 INDEX.md 和 RELATIONS.md 的维护策略

- INDEX.md 的目录树和设计表：在各批结束时统一更新，不要每写一个文件就更新一次
- RELATIONS.md 的关联边：每批结束时新增，避免频繁修改冲突

## 五、风险与建议

### 5.1 风险

| 风险 | 概率 | 影响 | 缓解措施 |
|------|:----:|:----:|---------|
| 各维度设计风格不一致 | 中 | 中 | 指定 02-经济与资源.md 为风格模板；架构师审查一致性 |
| 文件间依赖导致返工 | 中 | 高 | 先写底层（物品/功法），再写上层（战斗/任务）；批次间留评审缓冲 |
| 与现有剧本提取文件矛盾 | 低 | 高 | 所有设计基于已有的 1x 系列 extract 文件，不做推翻式改动 |

### 5.2 架构建议

1. **以 02-经济与资源.md 为模板**：该文件的格式（数据表 → 模型 → 消耗产出 → 平衡控制）可复用到所有维度
2. **「游戏角度」的定义**：每个要素回答三个问题——① 玩家如何与之交互（玩法）② 它的数值怎样运作（平衡）③ 它与其他要素如何联动（关联）
3. **INDEX.md 无需重建**：仅在现有设计表末尾追加新行，目录树追加新分支，可保留 git 历史清晰
4. **工作分支策略**：使用独立分支（如 `feature/game-design-expansion`）积累所有批次改动，再一次性合入主干
